# Knowledge Cognitive Cell v0.1 到 v0.2 进展说明与下一步计划

## 一、项目定位

本项目是 NSEAP 第 6 模块 Knowledge Cognitive Cell 的 MVP。它的目标不是简单做一个资料文件夹，而是验证一套可以持续沉淀课程、挑战、项目案例、提示词、FAQ 和 Agent 上下文的知识管理能力。

v0.1 主要验证“知识库应该如何展示和检索”。v0.2 开始验证“真实资料进入系统后，能否被保存、分类、搜索、编辑和管理”。

## 二、v0.1 原有能力

v0.1 是一个静态前端 Demo，主要实现了以下能力：

- 展示知识库首页、浏览页和知识详情页。
- 按课程、挑战、项目案例、提示词、FAQ、Agent 等类型筛选知识。
- 支持关键词搜索。
- 展示每条知识的 metadata，例如关键词、概念、技能、适用对象。
- 展示知识增长流程：场景、本体、流程、技能、评估、学习、知识增长。
- 展示知识之间的显式关系和 Agent 检索视角。

v0.1 的价值是把产品形态和知识模型先做出来，方便讨论模块定位和演示核心概念。但它的问题也比较明显：数据主要是固定在静态文件里，不能完整验证上传、入库、编辑、删除和模型配置这些真实工作流。

## 三、v0.2 已完成的主要升级

### 1. 从静态 Demo 升级为轻量前后端 MVP

v0.2 增加了 Node.js 后端服务。前端不再只依赖固定静态数据，而是通过 API 读取后端数据。

当前演示入口：

```text
http://127.0.0.1:8787/index.html
```

启动方式：

```bash
npm run dev
```

### 2. 增加本地 JSON 数据库

知识条目现在保存到本地数据库文件：

```text
data/knowledge-db.json
```

这意味着知识条目不再只是页面里的固定展示内容，而是可以被后端读取、写入、修改和删除。

### 3. 增加文件上传生成知识草稿

v0.2 增加了文件上传入口。用户可以上传资料，系统会：

- 保存原始文件到 `data/uploads/`。
- 自动生成一条 `draft` 状态的知识草稿。
- 把草稿写入 `data/knowledge-db.json`。
- 上传后可以在浏览页中搜索和查看该草稿。

这一步验证了“真实资料进入知识库”的基本链路。

### 4. 增加 metadata 在线编辑

上传或新增后的知识草稿，可以在详情页继续补充：

- 关键词
- 概念
- 技能
- 适用对象
- 审核状态

这对应正式知识库里的“资料进入系统后，需要被结构化整理和审核”的过程。

### 5. 增加 LLM 自动分类接入口

v0.2 在右上角设置面板中增加了模型配置能力：

- Base URL
- 模型名称
- API Key

配置后，系统可以在上传可读取文本的文件时调用 LLM，自动生成类型、摘要、关键词、概念、技能和适用对象。

当前能力边界：

- `.txt`、`.md`、`.csv`、`.json` 这类文本文件可以直接读取正文并尝试 LLM 分析。
- `.docx`、`.pdf` 目前会先保存文件并生成草稿，下一步需要接入正文抽取能力，才能让 LLM 真正读取里面的内容。

### 6. 优化 metadata-first 搜索逻辑

搜索不再只是模糊扫全文，而是优先匹配结构化 metadata：

- `keywords`
- `title`
- `type`
- `tags`
- `concepts`
- `skills`
- `audience`

这样搜索“挑战”时，会优先返回真正被标注为挑战或关键词包含挑战的条目，而不是随便返回正文里出现相关词的内容。

### 7. 增加知识删除功能

浏览页右侧详情区新增了删除按钮。为了避免误删，删除不是直接执行，而是：

```text
点击删除图标
-> 显示确认框
-> 点击确认删除
-> 后端删除数据库中的知识条目
-> 前端列表和搜索结果同步移除
```

当前 MVP 采用硬删除。正式版本建议升级为归档或回收站机制。

### 8. 增加设置面板和界面主题切换

右上角新增设置面板，包含：

- 模型配置
- API Key 配置状态
- 前端色系切换

这让演示时可以说明：当前系统已经预留模型接入和运行配置入口，后续可以继续扩展为更完整的知识工作台。

## 四、当前 v0.2 能演示的完整链路

当前 MVP 已经可以演示：

```text
上传资料
-> 后端保存文件
-> 生成知识草稿
-> 写入本地数据库
-> 前端读取数据库
-> 按 metadata 搜索
-> 打开知识详情
-> 补充 metadata
-> 必要时删除知识条目
```

这说明项目已经从 v0.1 的“静态展示”，升级为 v0.2 的“轻量可操作知识库”。

## 五、下一步计划

### 下一步 1：接入 docx / pdf 正文抽取

当前上传 Word 或 PDF 时，系统能保存文件，但还不能读取正文内容。下一步需要增加正文抽取能力，让系统可以从 `.docx` 和 `.pdf` 中提取文本，再交给 LLM 做分类和 metadata 生成。

这是最关键的下一步，因为真实学习资料和老师材料通常不会只用 `.txt` 或 `.md`。

### 下一步 2：优化 LLM 分类结果

接入 LLM 后，需要继续优化提示词和输出标准，让模型稳定返回：

- 知识类型
- 一句话摘要
- 关键词
- 概念
- 技能
- 适用对象
- 置信度

同时要保留人工修改能力，避免模型分类不准确时无法纠正。

### 下一步 3：把删除升级为归档机制

当前 MVP 是硬删除。正式版本更适合使用：

```text
active
draft
review
archived
deleted
```

这样删除不会马上丢失数据，而是先进入归档或回收站，方便恢复和审计。

### 下一步 4：增加知识审核流程

后续可以把知识条目分成：

- 草稿
- 待审核
- 已发布
- 已归档

这样可以更符合 Knowledge Team 的工作方式：资料先进入系统，再由人或 Agent 辅助整理，最后审核发布。

### 下一步 5：为 Agent 检索准备接口

当前页面已经展示 Agent 检索视角。下一步可以增加专门的 Agent API，例如：

```text
GET /api/agent/context?q=...
```

让 Agent 可以按关键词、技能、概念和任务场景检索知识，而不只是给人看。

### 下一步 6：准备 GitHub 提交与汇报材料

后续需要整理：

- MVP 代码
- 设计文档
- 进展说明
- 演示截图
- README 启动说明

然后提交到 fork 仓库对应目录，方便老师和同学 review。

## 六、汇报时可以这样总结

v0.1 解决的是“知识库长什么样、怎么组织知识”的问题。v0.2 解决的是“真实资料进入系统后，能不能被保存、分类、搜索、编辑和管理”的问题。

目前系统已经跑通了一个轻量知识库 MVP：可以上传资料、生成草稿、补充 metadata、按 metadata 搜索、配置 LLM、删除知识条目。下一步重点是接入 Word/PDF 正文抽取，并继续优化 LLM 自动分类能力。
